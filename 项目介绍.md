## 1. 一句话定位

**LambChat 是一个开源的、生产就绪的全栈 AI Agent 平台**——它不是一个「带工具的聊天框」，而是一整套把 Agent「真正干完活」所需的能力打包在一个可部署系统里：

- Agent 运行时（深度 Agent 图、子 Agent、思考模式、流式输出、人工审批、定时任务）
- 模型管理（多提供商、渠道路由、按角色可见）
- 工具与 MCP（系统级/用户级 MCP、密钥加密、按需加载、代码沙箱执行）
- 技能（Skill）与跨会话记忆（Memory）、技能市场、GitHub 同步、人设预设（Persona）
- 完整产品面（文件库、富预览、项目文件夹、会话分享、反馈、通知）
- 全端客户端（React 19 Web/PWA + Capacitor 移动端 + Tauri 桌面端）
- 生产基础设施（认证/RBAC、实时同步、可观测性、Docker/K8s 部署）

对应源码里的项目名是 `lamb-agent`（[pyproject.toml](pyproject.toml)），品牌名是 **LambChat**。

## 3. 整体架构（最重要的心智模型）

### 3.1 分层视图

```
┌──────────────────────────────────────────────────────────────┐
│  前端 frontend/  (React 19 SPA → Web/PWA + Capacitor + Tauri)   │
│    - SSE 接收聊天流；WebSocket 接收任务完成通知                    │
└───────────────┬──────────────────────────────────────────────┘
                │  HTTP / SSE / WebSocket
┌───────────────▼──────────────────────────────────────────────┐
│  src/api/   FastAPI 应用：路由 + 中间件 + 依赖注入                 │
│    - 34+ 路由模块；SPA 静态托管；SSE/WS 端点                       │
├──────────────────────────────────────────────────────────────┤
│  src/agents/   Agent 运行时（外层薄壳 graph）                     │
│    - search / fast / team 三个注册 agent + core 共享基础设施       │
├──────────────────────────────────────────────────────────────┤
│  src/infra/   基础设施服务层（约 40 个子模块）                     │
│    - 地基 / 领域数据 / AI 执行 / 编排 / 接入交付 五层              │
├──────────────────────────────────────────────────────────────┤
│  src/kernel/   内核：配置、schema、异常、共享类型                  │
└───────────────┬──────────────────────────────────────────────┘
                │
     MongoDB  │  Redis  │  PostgreSQL(可选)  │  S3  │  沙箱  │  第三方 SaaS
```

依赖方向：`kernel` ← `infra` ← (`api` + `agents`)。`kernel` 只放定义，不依赖任何上层。

### 3.2 ⭐ 核心洞察一：Agent 是「双层 Graph 嵌套」

这是理解整个运行时的钥匙。**LambChat 自己的 Agent 抽象只是一层薄壳**，真正的 Agent 循环由 `deepagents` 提供：

- **外层 graph**（LambChat 自建，[src/agents/core/base.py](src/agents/core/base.py)）：结构极简，`START → agent_node → END`。它唯一的职责是在那个节点里构造并驱动内层 graph。
- **内层 graph**（deepagents，在各 agent 的 `nodes.py` 里用 `create_deep_agent(...)` 构建）：这里才有真正的模型循环、工具调用、子 Agent、中间件栈、checkpointer。
- **执行流**：外层节点调用内层 `astream_events(version="v2")` → 每个原始事件交给 `AgentEventProcessor`（[src/infra/agent/events/processor.py](src/infra/agent/events/processor.py)）→ 经 `Presenter`（[src/infra/writer/present.py](src/infra/writer/present.py)）转成 SSE 事件推给前端，同时落库。

真正注册的 agent 只有 3 个（`core` 不是 agent，是共享基础设施）：

| Agent | 注册 ID | 排序 | 沙箱 | 定位 |
|---|---|---|---|---|
| SearchAgent | `search` | 1（默认） | ✅ | 搜索+执行，功能最全，支持沙箱 / auto_mode |
| FastAgent | `fast` | 2 | ❌ | 快速响应，纯内存 backend，无沙箱 |
| TeamAgent | `team` | 3 | ✅ | 团队路由，按角色分派子 Agent |

注册机制：`@register_agent("id")` 装饰器（[src/agents/core/base.py](src/agents/core/base.py)）把类塞进 `_AGENT_REGISTRY`；启动时 `discover_agents()`（[src/agents/__init__.py](src/agents/__init__.py)）通过 import 触发注册；`AgentFactory` 懒发现、懒实例化、带缓存。

### 3.3 ⭐ 核心洞察二：一次聊天请求的数据流

```
前端提交消息
  → POST /api/chat/stream           (src/api/routes/chat.py，后台执行、立即返回 run_id)
  → BackgroundTaskManager.submit    (src/infra/task/manager.py，本地 asyncio 或 arq 队列分发)
  → TaskExecutor.run_task           (src/infra/task/executor.py，建 session、装配 Presenter)
  → src/agents 外层 graph → 内层 create_deep_agent 执行
        · LLMClient.get_model       (src/infra/llm/client.py 取模型/路由/fallback)
        · mcp_global + 内置工具       (src/infra/tool/*，取工具、按需加载)
        · backend + sandbox          (src/infra/backend/deepagent.py 提供文件系统)
        · memory / skill / persona   (注入记忆、技能、人设)
  → Presenter 产出事件              (src/infra/writer/present.py)
  → dual_writer 双写                (src/infra/session/dual_writer.py：Redis Stream + MongoDB)
  → 前端 GET /api/chat/sessions/{id}/stream  (SSE 实时读取事件)
  → 任务完成后 WebSocket 推 task:complete 通知 (src/infra/websocket.py)
```

### 3.4 ⭐ 核心洞察三：两条贯穿全局的主线

1. **事件溯源 + 双写**：所有 Agent 输出统一经 `Presenter` 变成结构化事件 →
   `dual_writer` **即时写 Redis Stream**（保证 SSE 实时）+ **批量缓冲写 MongoDB**（保证不丢）→
   同一 `trace_id` 的事件聚合成一条 trace 文档（[src/infra/session/trace_storage.py](src/infra/session/trace_storage.py)）。
2. **分布式一致性**：配置/模型/工具/记忆等的跨实例同步统一走 **Redis Pub/Sub + 三级缓存（内存→Redis→DB）+ 分布式锁**，多路复用中枢是 [src/infra/pubsub_hub.py](src/infra/pubsub_hub.py)。这让系统具备多副本（K8s）部署能力。

### 3.5 ⭐ 核心洞察四：配置「数据库优先」

运行时配置的优先级是 **数据库 > 环境变量 > 默认值**。启动时 `initialize_settings()`（[src/kernel/config/service.py](src/kernel/config/service.py)）从 MongoDB 加载设置并覆盖全局 `settings` 对象；改设置会触发相关缓存重置（LLM 模型缓存、记忆后端、checkpoint）。因此**大部分配置（尤其是 LLM 模型和密钥）在部署后通过管理 UI 配置，而不是写死在环境变量里**。

## 5. 后端深度解析

### 5.1 API 层 `src/api/`

- **[src/api/main.py](src/api/main.py)**（约 840 行）：`create_app()` 装配 FastAPI 应用。要点：
  - **中间件顺序**（后加的先执行）：`RequestBodyLimitMiddleware`（8MB 请求体上限，上传路径豁免）→ `TracingMiddleware`（trace 上下文）→ `AuthMiddleware`（认证）→ `UserContextMiddleware`（用户上下文）→ 路由。
  - **路由注册**：34+ 个路由模块，前缀清晰（`/api/chat`、`/api/agents`、`/api/auth`、`/api/mcp`、`/api/skills`、`/api/sessions`…）。
  - **生命周期 `lifespan()`**：启动时初始化日志、默认角色、从数据库加载设置、并发初始化各 Storage 索引、启动分布式运行时监听器、后台预热 Agent 注册与模型缓存、回填会话搜索索引、启动飞书渠道；关闭时优雅释放所有连接池/沙箱/后台任务。**这是理解「系统启动都做了什么」的最佳单一入口。**
  - **前端托管**：生产模式直接托管构建好的前端静态资源（`./static`），并为 `/shared/{id}` 分享页做服务端 SEO 注入；开发模式重定向到 Vite dev server。
- **[src/api/deps.py](src/api/deps.py)**：依赖注入。`get_current_user_required` / `get_current_user_optional`（JWT 解析 + 45 秒进程内缓存 + 从 DB 动态取角色权限）、`require_permissions(...)`（RBAC 权限校验依赖）、WebSocket 认证。
- **[src/api/routes/](src/api/routes/)**：按领域拆分。最大的几个：`upload.py`(900)、`session.py`(853)、`mcp.py`(850)、`skill.py`(841)、`chat.py`(808)。`chat.py` 是聊天入口，`agent/` 子目录含 agent 列表/配置/模型路由。
- **[src/api/middleware/](src/api/middleware/)**：`auth.py`、`tracing.py`、`user_context.py`。

### 5.2 内核 `src/kernel/`

- **[src/kernel/config/base.py](src/kernel/config/base.py)**：`Settings` 类（pydantic-settings），**所有配置项的单一事实来源**。包含应用/LLM/MCP/会话/Redis/Mongo/Postgres/沙箱/S3/JWT/OAuth/邮件/记忆/推送等全部字段与默认值。JWT 密钥、MCP 加密盐、VAPID 密钥缺失时会自动生成（生产建议固定，见 [注意事项](#12-注意事项与已知问题)）。
- **[src/kernel/config/service.py](src/kernel/config/service.py)**：`initialize_settings()` / `refresh_settings()`——实现「数据库优先」覆盖，以及改设置后触发 LLM/记忆/checkpoint 缓存重置的联动。
- **[src/kernel/config/](src/kernel/config/)** 下 `_definitions_*.py`：设置项的元数据定义（分类、是否敏感、是否需重启）。
- **[src/kernel/schemas/](src/kernel/schemas/)**：核心数据模型（Pydantic）。重点看：
  - [agent.py](src/kernel/schemas/agent.py)：`AgentRequest`（聊天请求的完整字段：message、session_id、workspace_dir、启用/禁用工具技能、persona、附件、goal、auto_mode…）、`StreamEvent`、`AgentConfig`、`ToolInfo` 等。
  - [model.py](src/kernel/schemas/model.py)：`ModelConfig`（多提供商、per-model key/base 覆盖、fallback_model、vision profile）。
  - 其余：`message.py`、`mcp.py`、`role.py`、`permission.py`、`scheduled_task.py` 等。

### 5.3 基础设施层 `src/infra/`（后端最庞大的部分）

统一模式：绝大多数领域模块是 **Manager（业务门面）+ Storage（MongoDB 持久化）** 双文件。横向可分五层：

**① 地基层**
- [src/infra/storage/](src/infra/storage/)：统一存储抽象。`mongodb.py`/`redis.py`/`postgres.py`（三个 DB 客户端工厂，全层数据访问地基）、`s3/`（对象存储多后端：MinIO/阿里云/本地）、`checkpoint.py`（LangGraph checkpointer 工厂，按 `CHECKPOINT_BACKEND` 选 Mongo/Postgres/Memory）、`mongodb_store.py`（LangGraph BaseStore 的 Mongo 实现）。
- 横切支撑：`logging/`（`get_logger` + trace 上下文注入）、`utils/datetime.py`（时间工具，被极广泛引用）、`async_utils/blocking.py`（`run_blocking_io`，把同步 IO 挪出事件循环，高频使用）、`tracing/`（LangSmith）。
- [src/infra/pubsub_hub.py](src/infra/pubsub_hub.py)：共享 Redis pub/sub 多路复用中枢，分布式缓存失效的公共设施。
- [src/infra/runtime_services.py](src/infra/runtime_services.py)：**分布式运行时启停编排中枢**——`start/stop_runtime_services()` 集中拉起/关闭所有 pub/sub 监听器、arq 队列、调度器、记忆压缩 agent。**看懂它就看懂了 infra 各模块如何协作。**

**② AI 执行层（平台心脏）**
- [src/infra/llm/client.py](src/infra/llm/client.py)：多提供商模型路由。`PROVIDER_REGISTRY` 把 ~25 个 provider 映射到 anthropic/google/openai 三种协议；`get_model()` 做 provider 解析、per-model 覆盖、key 解析、LRU 实例缓存。
- [src/infra/agent/](src/infra/agent/)：**infra 级** Agent 组件（真正编排在 `src/agents`）——`events/`（把 deepagents 原始流转成前端事件）、`middleware/`（16 个中间件：重试+模型 fallback、prompt 缓存、工具拦截、提示注入、artifact 交付、子 agent 活动…）、`model_storage.py`（模型 CRUD，llm 层的数据源）。
- [src/infra/backend/deepagent.py](src/infra/backend/deepagent.py)：DeepAgent 文件系统后端三工厂（memory / persistent / sandbox），统一挂载 `/skills/` 与 `/memories/` 路由。
- [src/infra/sandbox/](src/infra/sandbox/)：代码沙箱。`session_manager.py` 管理「每用户一沙箱、跨 session 共享」的绑定关系；`base.py` 是 `SandboxFactory` + 三平台配置。
- [src/infra/tool/](src/infra/tool/)：工具管理。`mcp_client.py`/`mcp_global.py`（每用户 MCP 工具聚合 + Redis 分布式锁 + 缓存失效）、`tool_search.py`+`deferred_manager.py`（移植自 Claude Code 的按需工具发现）、以及一批工具实现（图像生成/分析、音频转写、reveal_file/project、human_tool 表单审批、scheduled_task 工具…）。
- [src/infra/mcp/](src/infra/mcp/)：MCP 配置存储层（系统级+用户级+偏好+策略+配额五个集合），`encryption.py` 用 Fernet 加密敏感字段。
- [src/infra/memory/](src/infra/memory/)：跨会话记忆。统一工具接口 `tools.py`（retain/recall/delete）、原生 MongoDB 后端 `client/native/`、后台压缩 agent `compaction_agent.py`。受 `ENABLE_MEMORY` 门控。
- [src/infra/writer/present.py](src/infra/writer/present.py)：统一 SSE 事件输出接口 + 按 trace 落库（前后端事件契约的后端侧）。

**③ 编排层**
- [src/infra/task/](src/infra/task/)：后台任务执行引擎。`manager.py`（提交入口，本地 asyncio 或 arq 分发，run_id 隔离）、`executor.py`（agent 真正运行处，串起 session/writer/agents）、`concurrency.py`（Redis Sorted Set 每用户并发限流+等待队列）、`recovery.py`/`heartbeat.py`/`cancellation.py`（崩溃恢复、心跳、分布式取消）。
- [src/infra/scheduler/](src/infra/scheduler/)：APScheduler 进程内定时任务。`runtime.py`（调度器）、`service.py`（CRUD 桥梁，支持 cron/interval/date）、`runner.py`（把触发接到任务管线）、`locks.py`（多实例分布式锁防重复触发）。受 `ENABLE_SCHEDULED_TASK` 门控。

**④ 领域数据层**（Manager+Storage 模式，约 20 个）
- [src/infra/auth/](src/infra/auth/)：`jwt.py`、`rbac.py`（内置 admin/user/guest 三种默认角色及权限清单）、`oauth.py`、`password.py`、`turnstile.py`。
- [src/infra/session/](src/infra/session/)：会话管理。`manager.py`（CRUD/fork/附件）、`trace_storage.py`（**事件溯源核心**）、`dual_writer.py`（**双写枢纽**）、`event_merger.py`（合并流式事件降数量）、`search_index.py`、`trace_event_chunks.py`（大 trace 分块，绕过 Mongo 16MB 限制）。

## 6. 前端深度解析

一套 React 代码库同时产出 **Web/PWA + Capacitor 移动端 + Tauri 桌面端**。

### 6.1 构建与入口
- **[frontend/package.json](frontend/package.json)**：React 19 + Vite 6 + TS + Tailwind；富内容渲染依赖（markdown/katex/mermaid/codemirror/sandpack/excalidraw/pdf/docx/xlsx）；`@microsoft/fetch-event-source`（SSE）；Capacitor/Tauri 插件；PWA（vite-plugin-pwa + workbox）；i18next。
- **[frontend/vite.config.ts](frontend/vite.config.ts)**：PWA 用 `injectManifest`（自定义 `src/sw.ts`）；`manualChunks` 手动分包；为浏览器 polyfill Node 模块；**dev proxy 把后端请求代理到 `http://127.0.0.1:8000`**，聊天流路由设 24 小时超时。

### 6.2 关键目录

### 6.3 跨端打包
- **[frontend/src-tauri/](frontend/src-tauri/)**：Tauri 2 桌面壳（Rust），带通知 + 自动更新（指向 GitHub Releases 的 `latest.json`）。

## 8. 部署与运维

三种部署形态，**共用同一个多架构镜像** `ghcr.io/yanyutin753/lambchat:latest`（amd64/arm64）：

| 形态 | 载体 | 定位 |
|---|---|---|
| **单机 Docker Compose** | [deploy/docker-compose.yml](deploy/docker-compose.yml) + `deploy.sh` | 开箱即用，单副本。自带 `redis:alpine` + `mongo:8.2.5` + lambchat 三个服务 |
| **Kubernetes** | [k8s/lambchat.yaml](k8s/lambchat.yaml) | 多副本生产。`replicas:2`、对称 pod 内嵌 arq worker、`LAMBCHAT_DISTRIBUTED_MODE=true`、**强制外部 S3**、健康探针、Secret 注入 |
| **裸机 / 源码** | `make dev`（后端）+ `pnpm dev`（前端），生产前置 [nginx/nginx.conf](nginx/nginx.conf) | 开发 / 自定义 |

- **镜像构建**：[Dockerfile](Dockerfile) 两阶段——Stage1 `node:20-alpine` 构建前端到 `dist`；Stage2 `python:3.12-slim` 用 uv 装依赖，并把前端 `dist` 复制为 `./static`（前后端一体、单容器）。运行时不需要 Node。
- **必需外部依赖**：**MongoDB**（主库）、**Redis**（缓存/流/pub-sub/队列/锁）。
- **可选依赖**：PostgreSQL（可选 checkpoint 后端）、S3（多副本强制）、Daytona/E2B（沙箱）、LangSmith/Resend/Turnstile/OAuth/Web Push 等 SaaS。
- **Nginx 关键点**：SSE 路由 `^/api/chat/sessions/[^/]+/stream$` 必须 `proxy_buffering off` + 长超时；WebSocket `/ws` 升级 + 86400s 超时；上传路径放大 body 上限。
- **CI/CD**（[.github/workflows/](.github/workflows/)）：`docker-build.yml`（多架构镜像推 GHCR）、`docs.yml`（VitePress → GitHub Pages）、`app-release.yml`（Tauri 桌面 + Capacitor 移动端全平台产物 → GitHub Release）、`lint.yml`（ruff/mypy/pytest/eslint/vitest/大文件检查）。
- **文档站点**：[docs/](docs/) 是 VitePress 中英双语站，`env/` 下 13 篇环境变量分类文档、`deploy/` 下 Docker/K8s/客户端打包指南。
- **运维脚本**：[scripts/](scripts/) 含沙箱快照/模板创建、trace 事件分块迁移与校验脚本。

## 12. 注意事项与已知问题

- **⚠️ `src/infra/service/` 疑似死引用**：[src/infra/service/__init__.py](src/infra/service/__init__.py) 从 `src.infra.service.milvus` 和 `src.infra.service.prometheus` 导入 `MilvusService`/`PrometheusService`，但目录下**只有 `__init__.py` 和 `base.py`**，这两个文件不存在。直接 `import src.infra.service` 会 `ImportError`。当前应用能正常运行，说明没有代码路径真的 import 它——这更像是被删文件后残留的死引用。**排查/清理相关代码前请留意，不要被它误导。**
- **密钥稳定性**：`JWT_SECRET_KEY`、`MCP_ENCRYPTION_SALT`、VAPID 密钥在缺失时会随机自动生成。单机没问题，但**多副本部署必须显式固定**这些值（否则各实例 token/加密不一致）。见 [backend-distributed-performance-audit.md](docs/backend-distributed-performance-audit.md)。
- **配置以数据库为准**：本地改 `.env` 后，如果对应设置已存进数据库，`.env` 不会生效（DB 优先）。改运行时行为优先用管理 UI 或直接改 DB。
- **文档轻微漂移**：`docs/en/deploy/docker.md` 写的镜像版本（`mongo:7`/`redis:7-alpine`）与实际 [deploy/docker-compose.yml](deploy/docker-compose.yml)（`mongo:8.2.5`/`redis:alpine`）不一致；`Makefile` 的 `docker-*` 目标在根目录用 `docker-compose`，但 compose 文件其实在 `deploy/` 下。**部署请以 `deploy/docker-compose.yml` 和 `deploy/deploy.sh` 为准。**
- **LLM 密钥不在 env**：模型和密钥通过部署后的 **Model Config UI** 配置，`.env` 里没有；基础启动路径不需要模型 key。
